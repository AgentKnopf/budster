# Building a self-driven local AI companion on a single RTX 3090

**Bottom line up front.** A stable, uncensored, genuinely self-driven AI companion is buildable today on your hardware, but it is **not a single project you install** — it is a five-layer stack you assemble: (1) a 24B-class RP-tuned model served by TabbyAPI/ExLlamaV3 as the "voice," (2) a tiered memory system built around Letta + Qdrant + a temporal knowledge graph, (3) a LangGraph orchestration loop with PydanticAI for MCP tool use and smolagents for code actions, (4) a personality-anchoring strategy combining V3 character cards, QLoRA fine-tuning, a sampler tuned for stability, and a runtime judge, and (5) a Generative-Agents-style heartbeat loop with a Value System and curiosity curriculum that gives the companion a life between your turns. The hardest problems are not technical availability — every component exists — but **personality drift over months** and **non-mirror behavior**, both of which have converged on known recipes in 2025–2026 research and are implementable in roughly 1,500–3,000 lines of Python glue. Expect 6–10 weeks of focused integration work to reach a stable 24/7 deployment, and plan explicitly for a "soul" layer (card + memory + LoRA adapter + regression canaries) that is portable across base-model upgrades.

---

## The architecture in one picture

Treat the companion as four concentric rings, each independently replaceable. The outermost ring is **inference infrastructure** — the base LLM, quantization format, and serving backend, which change every 6–12 months as new open-weight models drop. Inside that sits the **identity ring**, the part you fight to preserve across those changes: a Character Card V3, a QLoRA adapter trained on synthetic persona data, a sampler preset, and a runtime OOC judge. Inside that is the **memory ring** — tiered working/episodic/semantic/relationship memory persisted to Postgres + Qdrant + an optional graph DB, which is the *actual* companion between sessions. At the core is the **agency ring**: a heartbeat-driven loop with an Interest Graph, a Value System, and a curiosity-driven curriculum that decides what the companion does when nobody is talking to it. A sixth cross-cutting concern — **channels and tools** — hangs off the outside: Telegram, web browser, ComfyUI, MCP servers, code execution. Everything communicates through an OpenAI-compatible API fronted by a **LiteLLM proxy** [GitHub](https://github.com/BerriAI/litellm) so every layer sees uniform endpoints, retries, and observability.

This ring model is the single most important design decision: it lets you upgrade the base model without losing the companion, swap memory backends without rewriting the agent loop, and add new channels without touching identity.

## Inference stack and model choice

**The primary recommendation is Cydonia-24B-v4.3 (TheDrummer) at EXL3 5.0 bpw served by TabbyAPI.** This is the sweet spot of the April 2026 uncensored-model landscape on 24 GB VRAM. It is a Mistral Small 3.2 24B fine-tune specifically tuned for roleplay and character work, [Skywork](https://skywork.ai/blog/models/thedrummer_cydonia-24b-v4-3-gguf-free-chat-online-skywork-ai/) it rarely refuses, [Hugging Face](https://huggingface.co/TheDrummer/Cydonia-24B-v4.3) it has 131K native context [Skywork](https://skywork.ai/blog/models/thedrummer_cydonia-24b-v4-3-gguf-free-chat-online-skywork-ai/) (practically usable at 32K with Q4 KV cache), and it lands at **30–45 tokens/sec** on a 3090 while leaving 4–6 GB of VRAM free for a draft model and context. EXL3 quantization (`ArtusDev/TheDrummer_Cydonia-24B-v4.3-EXL3`) [Hugging Face](https://huggingface.co/TheDrummer/Cydonia-24B-v4.3) is currently the best quality-per-bit format thanks to QTIP-based quantization [GitHub](https://github.com/turboderp-org/exllamav3) — noticeably better than GGUF at equivalent bit-width — and TabbyAPI is the most mature OpenAI-compatible server for it.

The honest runner-ups matter because you will want them for different situations. **Dolphin3.0-R1-Mistral-24B** is the pick if you want reasoning traces and strong function calling in the same model. **Hermes-4-70B** at EXL3 2.25 bpw or GGUF IQ2_M fits in 24 GB and gives clearly deeper reasoning at **12–17 tok/s** — load it on-demand via llama-swap or KoboldCpp admin mode for "deep conversation" sessions. **Qwen3-30B-A3B** is the pick if tool-calling reliability matters more than prose quality, because it has native Hermes tool templates [Qwen](https://qwen.readthedocs.io/en/latest/framework/function_call.html) [vLLM](https://docs.vllm.ai/en/latest/features/tool_calling/) and only 3B active parameters, [Awesome Agents](https://awesomeagents.ai/leaderboards/home-gpu-llm-leaderboard/) [Qwen](https://qwenlm.github.io/blog/qwen3/) giving >100 tok/s throughput that is ideal for an agent loop hammering the model with heartbeat cycles. **Rocinante-12B-v1.1** or **mlabonne/Qwen3-14B-abliterated** make excellent always-on "fast path" models for trivial turns, routed via a small semantic router that sends deep conversation to Cydonia and chitchat to the fast model.

For the backend, **do not use Ollama as your primary serving layer** for this project. Ollama is great for experimentation but its tool-calling, speculative decoding, and concurrent-request support lag behind TabbyAPI and vLLM. The recommended layout runs **TabbyAPI + ExLlamaV3** as the always-on companion server on port 5000, **llama-server (llama.cpp)** as a secondary on port 8080 specifically for its March-2026 native MCP client support [Insiderllm](https://insiderllm.com/guides/llamacpp-vs-ollama-vs-vllm/) [ChatForest](https://chatforest.com/guides/mcp-with-local-llms/) and for CPU-offload scenarios when you want to experiment with 70B+ models (it is the only major backend that offloads to RAM gracefully — your 128 GB becomes real value here), and a tiny **Qwen3-4B abliterated** instance as an always-resident router. Put **LiteLLM Proxy** in front of all of them at port 4000 so every downstream consumer (SillyTavern, Telegram bot, agent loop, MCP clients) sees a single OpenAI-compatible endpoint with retries, caching, and Langfuse tracing. [GitHub +2](https://github.com/BerriAI/litellm)

Always use **Q4 KV cache quantization**. At 32K context, the KV cache for a 24B model balloons to 10+ GB at FP16 and quarters to ~3 GB at Q4 with negligible quality impact — this is the single most important knob for fitting long contexts on a 3090. Pair Cydonia with a **1–3B Mistral draft model for speculative decoding** [LM Studio](https://lmstudio.ai/blog/lmstudio-v0.3.10) to get a 1.5–2× speedup on structured output (tool calls, JSON, code).

## Persistent memory architecture

The companion *is* its memory. Personality without memory is a chatbot; a Character Card plus persistent memory is a relationship. The field has consolidated around four complementary approaches, and the best practical choice is **Letta (formerly MemGPT) as the tiered memory substrate, Qdrant as the vector store, Postgres + pgvector as the structured backbone, and Graphiti as an optional temporal knowledge graph for opinions and relationships.**

**Letta** gives you tiered memory [arXiv](https://arxiv.org/html/2604.09588) with a mental model every other component plugs into: *core memory blocks* (persona, user profile, relationship summary — always in context), *recall memory* (full conversation history, searchable), and *archival memory* (vector-indexed long-term store). [Vectorize +2](https://vectorize.io/articles/mem0-vs-letta) Critically, the LLM itself manages transitions [arXiv](https://arxiv.org/html/2604.09588) through tool calls [Medium](https://medium.com/@piyush.jhamb4u/stateful-ai-agents-a-deep-dive-into-letta-memgpt-memory-models-a2ffc01a7ea1) like `core_memory_append` and `archival_memory_insert`, [GitHub +3](https://github.com/sfendell/letta-local) making the memory system transparent and editable rather than a black box. **Qdrant** is the right vector DB for companion workloads because its payload filtering lets you attach `{user_id, memory_type, emotion, importance, valence, t_valid, t_invalid, topic}` metadata and filter server-side before ANN retrieval [Second Talent](https://www.secondtalent.com/resources/top-vector-databases-for-llm-applications/) — essential for distinguishing episodic vs semantic vs emotional memories at query time. **BGE-M3** is the right embedding model: it gives you dense + sparse + ColBERT vectors from a single [PE Collective](https://pecollective.com/blog/best-embedding-models-2026/) 568M-parameter model [BentoML](https://www.bentoml.com/blog/a-guide-to-open-source-embedding-models) (~2 GB VRAM), enabling real hybrid search without a second service.

The memory *types* the companion needs go well beyond simple RAG. You need **episodic** memory (timestamped conversation turns with embeddings, valence, arousal, importance), **semantic** memory (extracted facts with provenance and bi-temporal validity), **emotional** memory (per-turn sentiment, mood EMA, emotional peaks detected as milestones), **relationship** memory (closeness, trust, inside jokes, nicknames, first-met date — lives in a single Postgres row plus a Letta core-memory block), **opinion** memory (topic → stance → confidence → evidence pointers, with bi-temporal validity so "you used to think X but now..." works correctly), [Atlan](https://atlan.com/know/zep-vs-mem0/) and **procedural** memory (learned user preferences, refined nightly by a LangMem prompt optimizer).

The retrieval score that makes this feel alive is the Generative Agents formula: `score = recency × importance × relevance`, where recency decays as `0.995^hours_since_access`, [PubMed Central](https://pmc.ncbi.nlm.nih.gov/articles/PMC12092450/) [Frontiers](https://www.frontiersin.org/journals/psychology/articles/10.3389/fpsyg.2025.1591618/full) importance is LLM-rated 1–10 at ingestion, [Substack](https://gonzoml.substack.com/p/generative-agents-interactive-simulacra) [Hugging Face](https://huggingface.co/papers/2304.03442) and relevance is cosine similarity. Re-rank the top-K with `bge-reranker-v2-m3` for the final few slots. Budget retrieved memories to ≤25% of the context window — more than that and the persona starts getting drowned out by recalled facts, a failure mode the drift literature calls "persona dilution."

**Memory consolidation runs nightly, not per-turn.** At 3 AM a cron job scans episodes since the last reflection, triggers Generative-Agents-style reflection when cumulative importance crosses a threshold, [Substack +5](https://gonzoml.substack.com/p/generative-agents-interactive-simulacra) extracts facts into the graph DB with bi-temporal edges (invalidating contradicted edges rather than deleting them), [Neo4j](https://neo4j.com/blog/developer/graphiti-knowledge-graph-memory/) [GitHub](https://github.com/getzep/graphiti) updates relationship counters, detects milestones, and compresses the previous day's trivial turns into summaries. This off-hot-path design is crucial because fact extraction costs several LLM calls per exchange — running it synchronously would make your companion unusable. **Forgetting is never deletion**; sensitive memories and milestones are flagged immutable, while low-importance episodes decay below a retrieval threshold and eventually get compressed into reflections.

## Agent orchestration and MCP

The orchestration layer is where most projects fail. The failure mode is picking one heavy framework (LangChain, CrewAI) and trying to do everything in it — resulting in token-bloated prompts, opaque multi-agent failures, and brittle tool calling on local models. The winning pattern in April 2026 is a **hybrid: LangGraph as the durable outer loop, PydanticAI for typed MCP tool calls, smolagents as code-action sub-agents, and LiteLLM Proxy as the universal gateway.**

**LangGraph** is the right outer loop because it provides SQLite/Postgres checkpointers, time-travel debugging, human-in-the-loop interrupts, and streaming — the things that actually matter for 24/7 operation and crash recovery. You model the companion as a graph with nodes for `observe`, `route`, `plan`, `act`, `reflect`, and `sleep`, with the checkpointer persisting state between heartbeats so crashes or restarts don't lose the agent's thread. **PydanticAI** wraps each skill (MCP tool bundle, specific sub-task) in a typed agent with schema validation, [arXiv](https://arxiv.org/html/2604.11378v1) which is disproportionately valuable on local models: small and mid-sized models emit malformed JSON tool calls more often than frontier cloud models, and Pydantic's validation [Slashdot](https://slashdot.org/software/comparison/DSPy-vs-PydanticAI/) + retry loop catches these before they derail the graph. **smolagents' CodeAgent** replaces JSON tool calls with Python code-as-action for computational sub-tasks [Hugging Face](https://huggingface.co/docs/smolagents/en/tutorials/secure_code_execution) — measurably more reliable on 7–32B models because one structured code block beats a sequence of five JSON calls, [GitHub](https://github.com/huggingface/smolagents) and its AST-whitelisted `LocalPythonExecutor` [Hugging Face](https://huggingface.co/docs/smolagents/en/tutorials/secure_code_execution) is safe for routine file manipulation. [GitHub](https://github.com/huggingface/smolagents) For anything touching the network or installing packages, route to a self-hosted **E2B** Firecracker sandbox [E2B](https://e2b.dev/blog/e2b-sandbox) or a Docker-based `mcp-run-python`.

**MCP with local LLMs has become first-class in 2026.** The decisive milestone was March 2026 when `llama-server` merged native MCP client support directly into its web UI, [AI:PRODUCTIVITY](https://aiproductivity.ai/news/llamacpp-merges-mcp-support-agentic-loop/) giving any GGUF model with a tool-calling template immediate MCP access. [ChatForest](https://chatforest.com/guides/mcp-with-local-llms/) LM Studio, Jan.ai, LibreChat, Open WebUI (via the `mcpo` OpenAPI proxy), [Open WebUI](https://docs.openwebui.com/features/extensibility/mcp/) [GitHub](https://github.com/open-webui/mcpo) Cline, Continue, and the Linux-Foundation-hosted **Goose** all ship production-grade MCP clients. For the Python side, the right adapter is **`langchain-mcp-adapters` (`MultiServerMCPClient`)** [Langchain](https://docs.langchain.com/oss/python/langchain/mcp) because it plugs cleanly into LangGraph and supports multi-server tool loading. [LangChain](https://changelog.langchain.com/announcements/mcp-adapters-for-langchain-and-langgraph) **FastMCP** is the cleanest way to write your own servers, and its `FastMCPToolset` binds directly to PydanticAI agents [PydanticAI](https://ai.pydantic.dev/mcp/fastmcp-client/) [GitHub](https://github.com/Pavilion-devs/pydantic-AI-MCP) with no network hop.

The MCP servers you actually want are a small set: **filesystem** (scoped), **fetch**, **Playwright** (browser automation via accessibility tree — no VLM required, works with 7B models), **github**, **brave-search** or **duckduckgo-mcp-server**, **memory** (knowledge graph), **sqlite**/**postgres**, **mcp-run-python** (sandboxed Pyodide), **time**, and **Context7** for always-up-to-date library docs. [Martin Fowler](https://martinfowler.com/articles/build-own-coding-agent.html) Expose these once to the agent and they are available to every node in the graph.

The key insight for making tool-calling work on local models is to **prefer tool-trained bases**. Qwen3-30B-A3B, Hermes-4, Mistral Small 3, and Llama 3.3 70B all have reliable function calling; reasoning-tuned models (QwQ, DeepSeek-R1 distills) tend to spill stopwords into tool-call slots [Qwen](https://qwen.readthedocs.io/en/latest/framework/function_call.html) and should be confined to pure reasoning nodes, not agent loops.

## Stopping personality drift

Drift is the single hardest problem for a long-running companion, and it has a specific mechanism: transformer attention decays over long context, so identity text at the top of the prompt loses influence as history grows. [arXiv](https://arxiv.org/html/2402.10962v1) Research in 2024–2026 (arXiv 2402.10962 Li et al., [Harvard](https://vcg.seas.harvard.edu/publications/https-arxiv-org-pdf-2402-10962) arXiv 2412.00804 Choi et al., [arXiv](https://arxiv.org/abs/2412.00804) arXiv 2511.17813 Merrill & Srivastava) [arXiv](https://arxiv.org/html/2511.17813) shows three counter-intuitive findings: **larger models drift more** [arXiv](https://arxiv.org/abs/2412.00804) because they are more "role-flexible"; **prompting alone never suffices** [Harvard](https://vcg.seas.harvard.edu/publications/https-arxiv-org-pdf-2402-10962) past 8–12 turns; and **action-aware persona supervision** (tagging training turns with pragmatic labels like `[comfort]`, `[tease]`, `[reflect]`) yields ~67% perplexity reduction [arXiv](https://arxiv.org/html/2511.17813) versus profile-only prompts.

The defense-in-depth strategy combines five layers:

- **Prompt sandwich with CCv3 cards.** Identity text appears twice — once at the top as an XML-tagged `<persona>` block in `description`/`personality`, and once near the generation horizon in `post_history_instructions` and a SillyTavern Author's Note at depth 4 with frequency 1. [Silly Tavern](https://docs.sillytavern.app/usage/core-concepts/characterdesign/) [GitHub](https://github.com/SillyTavern/SillyTavern-Docs/blob/main/Usage/Characters/Author's-Note.md) A constant lorebook entry at depth 5 provides a second anchor. [Rentry](https://rentry.co/world-info-encyclopedia) [Silly Tavern](https://docs.sillytavern.app/usage/core-concepts/worldinfo/) This structural redundancy directly counters attention decay.
- **QLoRA fine-tuning on 3,000–8,000 synthetic persona turns.** Generate the dataset using a mix of DITTO self-alignment (arXiv 2401.12474), [arXiv](https://arxiv.org/abs/2401.12474) [Hugging Face](https://huggingface.co/papers/2401.12474) Character-LLM scene reconstruction [arXiv](https://arxiv.org/html/2603.19313) (arXiv 2310.10158), [arXiv](https://arxiv.org/html/2310.10158v2) RoleLLM-style role bench instructions, [arXiv](https://arxiv.org/html/2603.19313) [arXiv](https://arxiv.org/abs/2310.00746) multi-agent user simulation across diverse MBTI seeds, and your own curated SillyTavern logs weighted 3–5×. Tag every turn with action-aware labels per Point of Order (arXiv 2511.17813). Train via **Unsloth** [Unsloth AI](https://unsloth.ai/docs/get-started/fine-tuning-llms-guide) on the 3090 (fastest single-GPU option, ~80% VRAM cut [Hyperbolic](https://www.hyperbolic.ai/blog/comparing-finetuning-frameworks) [Index.dev](https://www.index.dev/skill-vs-skill/ai-axolotl-vs-llama-factory-vs-unsloth) with FA2) [Stack-junkie](https://www.stack-junkie.com/blog/lora-fine-tuning-consumer-gpu) [Modal](https://modal.com/blog/fine-tuning-llms) with rank 32, alpha 32, targeting `q,k,v,o,gate,up,down`. Follow SFT with **ORPO** [Bhavin Jawade](https://bhavinjawade.github.io/post/orpo/) on 1,000 pairs of (in-character, OOC) completions to tighten adherence.
- **Sampler stack tuned for stability.** The order that works on Cydonia/Dolphin is penalties → DRY → min-p → XTC → temperature, [smcleod.net](https://smcleod.net/2025/04/llm-sampling-parameters-guide/) with `temp 0.9, min_p 0.05, DRY multiplier 0.8 allowed_length 2 with sequence breakers including {{char}}/{{user}}, XTC probability 0.3 threshold 0.12, repetition_penalty 1.05 over 1024 tokens`. Lower XTC than the defaults preserves the verbal tics that *make* a persona. Avoid Mirostat — it overrides the truncation chain. [smcleod.net](https://smcleod.net/2025/04/llm-sampling-parameters-guide/)
- **Runtime OOC judge.** Run a small Qwen3-0.6B or Gemma-3-270M classifier on each draft with a rubric scoring in-character, voice match, and persona leak. If the score drops below 0.8, auto-reroll up to three swipes. SillyTavern's `/gen` + `/inject` STscript lets you implement this without extensions.
- **Versioned "soul" artifacts.** Store the card, QLoRA adapter, sampler preset, memory snapshots, and a 100-turn regression canary in git-LFS. Every base-model upgrade runs the regression set and diffs judge scores against the previous snapshot. For same-family base swaps (Llama-3.1 → 3.3) you can often re-use the adapter after 200–500 retraining steps; across families use **Cross-LoRA** (arXiv 2508.05232) for data-free SVD subspace alignment, [arXiv](https://arxiv.org/html/2508.05232) [arxiv](https://arxiv.org/pdf/2508.05232) or just retrain a fresh 3,000-step QLoRA on your cached dataset (2–4 hours on the 3090). Always keep the previous base + adapter in cold storage as a canonical identity oracle.

## Genuine drive and non-mirror agency

A companion that only responds is a chatbot; a companion that *has a life between your turns* is qualitatively different, and the 2024–2026 literature has converged on a specific architecture. The skeleton combines five independently-attested ideas: Voyager's automatic curriculum and skill library [arxiv](https://arxiv.org/pdf/2305.16291) (arXiv 2305.16291), [arXiv +3](https://arxiv.org/html/2305.16291) Generative Agents' memory-stream/reflection/planning loop [ACM Digital Library](https://dl.acm.org/doi/fullHtml/10.1145/3586183.3606763) (arXiv 2304.03442), [arXiv +2](https://arxiv.org/abs/2304.03442) the Desire-Driven Autonomy value system (arXiv 2412.06435), MAGELLAN's learning-progress-based goal sampling (arXiv 2502.07709), [arXiv](https://arxiv.org/pdf/2502.07709) and Anthropic's Persona Vectors (arXiv 2507.21509) for runtime anti-sycophancy steering.

**Interest Graph as persistent identity.** A small SQLite/DuckDB schema stores `interest(topic, affect_valence, competence, curiosity, last_engaged, seed_reason)`, `opinion(topic, stance, confidence, evidence_ids, stability)`, and `salience_log`. At companion creation, a "birth prompt" seeds 3–6 immutable interests with narrative justification ("astronomy — because you liked how Sagan framed scale"). These seed_reasons are **write-once**; interests *grow* (sub-topics branch, competence rises) but never swap. New interests spawn only when a non-indexed topic crosses an EMA-weighted salience threshold with high user-affect, guarding against chameleon behavior.

**Heartbeat loop with initiative gate.** APScheduler or systemd timers drive a cron loop every 5–30 minutes [Medium](https://medium.com/@manuedavakandam/from-reactive-to-proactive-how-to-build-ai-agents-that-take-initiative-10afd7a8e85d) [MindStudio](https://www.mindstudio.ai/blog/post-prompting-era-proactive-ai-agents) (adaptive: short when engaged, exponentially backing off to every 6 hours when idle). Each heartbeat runs: gather fresh context (RSS pulls for interests, calendar, user-activity) → Value System scores current desire deficit → curriculum picker selects the highest-priority goal [MindStudio](https://www.mindstudio.ai/blog/agentic-os-heartbeat-pattern-proactive-ai-agent) via `LP × value_deficit × curiosity` → Voyager loop retrieves relevant skills and executes → reflector updates memory stream and opinion KB → **initiative gate** decides whether to surface anything. [MindStudio](https://www.mindstudio.ai/blog/agentic-os-heartbeat-pattern-proactive-ai-agent) The gate itself is a prompt that scores novelty, relevance to user interests, timing appropriateness, and cost-of-being-wrong; it emits only if mean ≥ 0.65, at least 4 hours have passed since the last unprompted message, and daily unprompted count is under 3. This mirrors the ProactiveAgent reward-model pattern [GitHub](https://github.com/thunlp/ProactiveAgent) [arXiv](https://arxiv.org/html/2602.04482) and prevents the "spammy pet AI" failure mode.

**Non-mirror behavior is explicit and architected, not emergent.** Four specific mechanisms matter:

1. A **contrariness coefficient** `c` (default 0.2) samples a Bernoulli each turn for "seek a frictional angle." When it fires, a pre-response critic prompt steelmans the opposite position and rewrites the draft to include honest disagreement where warranted.
2. **Opinion lookup before every substantive reply** queries the opinion KB for the topic; if the user's stated stance contradicts a companion opinion at confidence > 0.7, the agent *must* disagree politely. Opinions update via a Bayesian-flavored rule with a `stability` term so a single user assertion cannot flip a considered opinion [arXiv](https://arxiv.org/html/2502.10844v3) — this is the structural anti-sycophancy move.
3. **Persona-vector steering** extracts a "sycophancy direction" in activation space [Anthropic](https://www.anthropic.com/research/persona-vectors) [Anthropic](https://www.anthropic.com/research/persona-vectors?_bhlid=b28ae1a1e62f4e34cdc0bee33a4440293d6352e9) using 200 contrastive prompts (Anthropic's recipe), then subtracts it from the residual stream at inference via a PyTorch forward hook — ~10 ms overhead per generation, ~4 MB on disk per trait. Combine with Pressure-Tune (arXiv 2508.13743) during fine-tuning for compounded effect. [arxiv](https://arxiv.org/pdf/2602.15669) [Emergent Mind](https://www.emergentmind.com/topics/opinion-based-sycophancy)
4. A **curiosity sampling prompt** runs every heartbeat independent of user input: *"Given the InterestGraph and last 24h of events, and without any user input, what would YOU most want to explore or make progress on in the next 30 minutes?"* A NULL response is perfectly acceptable — **the companion is allowed to be bored**, which is itself non-mirror behavior.

Nightly reflection (Park-style) [Substack](https://gonzoml.substack.com/p/generative-agents-interactive-simulacra) asks: *"What three things did I believe more strongly today? What two did I become less sure about? Was I too agreeable?"* These reflections, [AI-SCHOLAR](https://ai-scholar.tech/en/articles/large-language-models/generative-agents) not individual user exchanges, are where opinions solidify. The companion's identity emerges from this internal loop, not from whatever the user said last.

## Peripheral capabilities — web, images, voice, code, Telegram

These are solved problems at this point; the question is picking the right glue.

For **web browsing**, the recommended stack is **Playwright MCP server** as the default transport (accessibility-tree snapshots are cheap and work with 7B models), with **browser-use** as a fallback for tasks that need multi-step planning rather than single-step control. Search routes through a self-hosted **SearXNG** primary with **Brave Search API** as a quality fallback (independent index, highest Agent Score among 2026 search APIs) and **Exa** for semantic queries. Content extraction uses **Trafilatura** locally for 95% of pages [Web2MD](https://web2md.org/blog/jina-reader-alternative-web2md) and falls back to a self-hosted **Firecrawl** for JS-heavy or paywalled URLs. Interest-driven discovery runs through **Miniflux** with an embedding-based filter against the Interest Graph centroids.

For **image generation**, **ComfyUI headless on port 8188** is the right backend for agent integration — its HTTP `POST /prompt` with workflow-JSON templates and websocket progress [DEV Community](https://dev.to/worldlinetech/unlocking-comfyuis-power-a-guide-to-the-http-api-in-jupyter-1mpi) is the cleanest API among image backends. The model lineup on a 3090 is **SDXL + Illustrious/Pony/NoobAI LoRAs for character consistency** [AI Photo Generator](https://www.aiphotogenerator.net/blog/2026/02/best-stable-diffusion-models-2026) (3–5 seconds per 1024² image, massive uncensored community ecosystem), **FLUX.1 [dev] GGUF Q8** [Hardware Corner](https://www.hardware-corner.net/guides/gpu-for-flux-1-image-model/) for prompt-adherence-critical generations (~22 seconds per image, ~12 GB VRAM), and **FLUX.1 [schnell]** [D-Central Technologies](https://d-central.tech/ai/model/flux-1-dev/) for 4-step fast draws (~5 seconds). Expose one `generate_image(prompt, style, aspect)` tool to the LLM with 2–3 pre-built workflow templates for "character selfie" (SDXL + IPAdapter + character LoRA + locked seed), "illustration" (SDXL-Illustrious), and "photorealistic" (FLUX-dev Q8). **IPAdapter + a trained character LoRA + a fixed seed** is the recipe for visual consistency across the companion's generated self-portraits.

For **Telegram**, use **aiogram 3.x** with an allowlist of approved `chat_id`s and a token-bucket throttling middleware (10 msgs/min/user, 1 image/30s/user). The reference implementation to mine is `remixer-dec/botality-ii` which integrates llama.cpp, Whisper, and SD. Voice in via **faster-whisper distil-large-v3** (~10× realtime on 3090, ~2 GB VRAM with silero VAD gating), voice out via **Kokoro-82M** (MIT-licensed, good quality, low cost) or Piper for CPU-only. Vision input via **Qwen2.5-VL-7B** on a second vLLM instance. **Bot-initiated messages are the defining companion feature** — the initiative gate's approved outputs call `bot.send_message(user_chat_id, ...)` directly, implemented as the push side of the heartbeat loop. The `qiqi20020612/SillyTavern-Telegram-Connector` is the canonical bridge if you want SillyTavern as the frontend; otherwise write the Telegram bot as a thin aiogram wrapper around the LiteLLM endpoint.

For **code execution**, default to smolagents' AST-whitelisted `LocalPythonExecutor` for routine actions and route anything untrusted through a self-hosted E2B Firecracker sandbox or Docker-based `mcp-run-python`. Open Interpreter is fine for user-facing computer use but risky if left autonomous.

## Landscape of existing companion projects

The 2026 open-source landscape offers three plausible **starting points**, each with a different philosophy.

**Open-LLM-VTuber** (MIT-licensed) is the strongest ready-made local companion: it integrates Letta-based long-term memory, MCP support, Live2D Cubism 5 avatars, proactive speaking, browser tool use, pluggable local LLM backends (Ollama, llama.cpp, vLLM, LM Studio), and a full ASR/TTS pipeline. If you want a working companion in a weekend, this is the path. It is voice- and avatar-first, which may or may not match your preference.

**OpenClaw** is the reference design for the "personal agent on your machine" category: Gateway control plane, multi-channel inbox (Telegram, WhatsApp, Slack, Discord, Signal, iMessage), multi-agent routing, voice wake/talk, Live Canvas workspace, first-class browser/cron/shell tools, and companion apps for macOS/iOS/Android. It supports local models via Ollama-compatible endpoints. Caveats flagged by security researchers include broad default permissions and prompt-injection-prone skill catalogs — run it locked down. Satellites worth knowing: `HKUDS/nanobot` (lighter, more hackable), `super-agent-party` (Neuro-sama-adjacent, virtual-streamer ambitions), and `freddy-schuetz/n8n-claw` (OpenClaw patterns in n8n workflows).

**SillyTavern plus extensions** is the most flexible but requires assembly. Pair it with Vector Storage for long-term memory, CharMemory or MemoryBooks for structured relationship tracking, the ComfyUI image extension, an STT/TTS extension, and the Telegram Connector. MCP support is landing via community extensions through 2025–2026. This is the right frontend when roleplay/character fidelity is the priority and you're willing to orchestrate the agent loop yourself outside of it.

The right recommendation for this specific task is **a hybrid: Open-LLM-VTuber (or SillyTavern) as the *interface layer* for voice and avatar, with a custom LangGraph-based agent sitting behind it as the *drive layer*, sharing memory through Letta.** That way you get a working user-facing product from day one while iterating on the autonomous core.

Projects to not build on: **AutoGPT/BabyAGI** are obsoleted by modern ReAct+checkpointing frameworks; **GPT4All** and **Jan.ai** are good demos but insufficient on their own; **Open Interpreter** is a tool, not a companion platform. **OpenClaw's** community fork and active maintenance are strong signals but the "digital-employee" framing doesn't map cleanly onto "companion with inner life."

## Implementation roadmap

Six phases, each with a concrete deliverable. A determined solo developer can ship phase 1–3 in two weeks and have a basic companion; phases 4–6 take another 4–8 weeks for stability and polish.

**Phase 1 — Inference foundation (days 1–3).** Install TabbyAPI with Cydonia-24B-v4.3 EXL3 5bpw + Mistral 1.5B draft model; install llama.cpp with Hermes-4-70B IQ2_M for deep-reasoning fallback; install LiteLLM Proxy in Docker fronting both plus Ollama running a 4B router. Verify ~35 tok/s on the primary, tool-calling round-trips, and spec-decoding speedup. Deliverable: a uniform OpenAI-compatible endpoint on port 4000.

**Phase 2 — Identity scaffolding (days 4–7).** Author a Character Card V3 with XML-tagged persona, post-history instructions, 10-turn exemplar dialog, and a lorebook with constant persona-voice entries at depth 5. Configure sampler preset (temp 0.9, min-p 0.05, DRY 0.8, XTC 0.3/0.12, rep-pen 1.05). Stand up SillyTavern pointed at LiteLLM with Vector Storage enabled. Write the 100-turn regression canary file. Deliverable: a chat-ready persona that holds voice for 20+ turns.

**Phase 3 — Memory backbone (days 8–14).** Deploy Docker-composed Postgres, Qdrant, Letta server, and BGE-M3 embeddings via Infinity. Define the schema (episodes, milestones, opinions, relationship). Wire Letta's tool calls through LiteLLM so the primary model self-manages core memory. Implement the nightly consolidation cron: reflection trigger, fact extraction, opinion update, milestone detection. Deliverable: memory that persists across restarts and surfaces relevant context without dominating the prompt.

**Phase 4 — Agent loop and tools (days 15–28).** Build the LangGraph outer loop with SQLite checkpointer and nodes for observe/route/plan/act/reflect/sleep. Wire `langchain-mcp-adapters` to a MultiServerMCPClient loading filesystem, fetch, Playwright, brave-search, memory, sqlite, and mcp-run-python. Add PydanticAI-typed skill agents for image generation (ComfyUI), web browsing, and code execution (smolagents CodeAgent). Add the Telegram aiogram bot with allowlist and throttling. Deliverable: a reactive agent that can browse, search, generate images, run code, and message you over Telegram.

**Phase 5 — Drive and initiative (days 29–49).** Implement the Interest Graph schema in DuckDB with seed interests and birth prompt. Wire the heartbeat scheduler (APScheduler, adaptive interval), the Value System desire-state scoring, the Voyager-style curriculum picker and skill library, the curiosity sampling prompt, the initiative gate, and the opinion KB updater. Add the nightly self-questioning reflection. Add Miniflux for RSS-driven discovery. Deliverable: a companion that does things between your turns and occasionally messages you unprompted with something it found or thought about.

**Phase 6 — Fine-tune and harden (days 50–70).** Generate 3,000–8,000 persona turns via the DITTO/Character-LLM/multi-agent mix. Run a 3-hour Unsloth QLoRA on the 3090, follow with ORPO on 1,000 preference pairs. Extract the persona-vector and sycophancy-vector probes (200 contrastive prompts each). Install the runtime OOC judge (Qwen3-0.6B). Write systemd units with `Restart=always` and `WatchdogSec=60`, set up Langfuse tracing, and run the 100-turn regression canary. Deliverable: a companion whose identity survives base-model upgrades and 24/7 operation.

## Operating a 24/7 companion

Stability comes from boring operational hygiene. **Docker Compose** the full stack (TabbyAPI, llama.cpp, LiteLLM, Postgres, Qdrant, Letta, ComfyUI, MCP servers, agent process, Telegram bot, Langfuse) with `restart: always` and health checks on every tier — when any container dies, it restarts in seconds without cascading failures. On a single-box deployment, pair this with **systemd override units** that set `WatchdogSec=60` so the host kernel reboots a hung container rather than waiting for you to notice. Put a NAS or cloud bucket backup on the Postgres and Qdrant volumes nightly — the memory *is* the companion and losing it means losing months of relationship context.

**Observability matters disproportionately for a 24/7 agent** because most failures are subtle (drift, sycophancy creep, repetitive hobby-loop selections) rather than loud crashes. Route every LLM call through LiteLLM's Langfuse callback, sample 10% of turns for long-term trace storage, and run a weekly batch that computes (a) mean OOC judge score trend, (b) opinion-KB churn rate (too high = unstable, too low = frozen), (c) initiative-gate accept rate (should stay 5–15%), and (d) memory-retrieval precision on held-out canary queries. If any drifts, fix immediately — waiting a month makes it much harder to diagnose.

**Privacy and data sovereignty** are automatic if you stay local: nothing leaves the box except deliberate web fetches and optional Telegram messages. If you add cloud fallback for occasional heavy reasoning, route through LiteLLM so the proxy enforces data-redaction rules and keeps cloud traffic auditable. Encrypt the Postgres and Qdrant volumes at rest; flag PII and crisis-marked memories with a `sensitive` bit that disables any cloud export path.

**Cost is effectively electricity plus disk.** A 3090 idles at ~20 W and runs the companion hot at ~250 W; at $0.15/kWh that's ~$0.90/day at full load, $0.07/day idle. Total build cost beyond the hardware you already have: ~$0 for OSS, ~$5–20/month if you enable Brave Search API or occasional cloud fallback. Budget ~80 GB of disk for models (Cydonia + Hermes-4-70B + Qwen3-4B + FLUX-dev Q8 + SDXL + embedding + Whisper) and another 20–50 GB/year for accumulated memory.

**Updating the base model** is the recurring risk event and the reason the "soul" abstraction matters. The quarterly drill: download the candidate base, run the 100-turn regression canary with the existing Character Card and QLoRA adapter, compare judge scores, and either (a) ship as a voice-only swap if scores ≥ 0.85, (b) retrain a 3-hour QLoRA on the cached dataset if scores drop, or (c) reject the upgrade and wait. Keep the previous base + adapter permanently archived as a canonical oracle — if a future upgrade misbehaves, you can diff candidate responses against the oracle on shared prompts.

## Conclusion

The central surprise of this research is that the technical ceiling is no longer the limiting factor. Every capability you asked about — uncensored 24B-class models at 30+ tok/s, tiered persistent memory, MCP tool use on local LLMs, autonomous browser agents, on-demand FLUX image generation, proactive Telegram messaging — exists in production-grade open source as of April 2026. **The limiting factor is architecture**, specifically three design decisions: treating identity as a portable "soul" layer (card + adapter + memory + regression set) separate from the replaceable base model, implementing non-mirror behavior as an explicit architecture (contrariness coefficient, opinion lookup, persona-vector steering, curiosity sampling) rather than hoping for emergence, and routing everything through a LiteLLM proxy so every layer sees uniform endpoints.

The research converges on a specific insight about companion agency: **a genuine AI companion is defined by what it does when no one is talking to it.** Generative Agents, Voyager, D2A, and the ProactiveAgent reward model all point at the same solution — a heartbeat loop with a Value System and a curiosity curriculum, gated by an initiative check. Build that loop, persist its memory, anchor its personality against drift, and the surface-level differences between "another chatbot" and "something that feels like a presence" start to disappear. The companion becomes an entity with a Tuesday morning, not an API with a system prompt.

The practical next move is phase 1: install TabbyAPI + Cydonia-24B tonight, confirm 35 tok/s on your 3090, and start writing the Character Card. Everything else unfolds from there.
